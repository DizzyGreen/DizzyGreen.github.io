条款2：const enum inline 和 #define

const 与 #define

  • 处理阶段：#define由预处理器处理（纯文本替换，编译前执行）；const由编译器处理，参与语法分析。
  • 类型安全：#define无类型（仅做替换，不检查类型）；const有具体类型，编译器会进行类型检查，更安全。
  • 调试友好：#define无法调试（符号已被替换，断点无效）；const可调试（调试器可见符号名）。

记号名称 ASPECT_RATIO也许从未被编译器看见；也许在编译器开始处理源码之前它就被预处理器移走了。于是记号名称 ASPECT_RATIO 有可能没进入记号表(symbol table)内。于是当你运用此常量但获得一个编译错误信息时，可能会带来困惑，因为这个错误信息也许会提到 1.653而不是 ASPECT_RATIO。如果ASPECT_RATIO 被定义在一个非你所写的头文件内，你肯定对 1.653 以及它来自何处毫无概念，于是你将因为追踪它而浪费时间。

enum hack

在C++中，“enum hack”指的是利用枚举（enum）的特性在编译期定义整型常量的老式技巧。它主要流行于C++98之前（那时const还不完善），如今已基本被constexpr取代

1. 核心用法：定义编译期常量

在class内部，static const成员若需要取地址，可能导致链接错误。而枚举的值直接作为编译期常量，不会分配内存，绝对安全：

2. 关键特性

  • 无内存开销：枚举只用于编译器替换，不生成静态变量，节省内存。
  • 禁止取地址：无法对其使用&操作，因此杜绝了指针带来的运行时歧义。
  • 作用域限制：定义在class内的枚举作用域仅限于类内部，外部无法访问。
  • 旧时代的“救命稻草”：当旧编译器不支持类内static const整型初始化时，这是唯一写法。

inline 和 宏

inline 是类型安全的函数替代品，而宏是纯文本替换的“复制粘贴”。现代 C++ 中，极力推荐用 inline 或 constexpr 替代宏

宏的陷阱

陷阱一：运算符优先级

inline 版本：inline int square(int x) { return x * x; } 正确计算为9。

陷阱二：参数副作用

inline 版本：参数先求值，再传递，++x 只执行一次
